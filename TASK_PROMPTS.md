# HiveOps 蜂场养殖管理平台 - 任务 Prompt 集

> 基于 `React + FastAPI + Prometheus + PostgreSQL` 的 HiveOps 平台规划，覆盖五种任务类型：
> - **[Feature]**：在已有功能基础上的平滑扩展（20 道）
> - **[0-1]**：简单完整模块的从零构建（20 道）
> - **[Test]**：单测 / 集成测试 / 压测（3 道）
> - **[Refactor]**：性能优化 / 可读性 / 解耦（2 道）
> - **[DevOps]**：环境配置 / 依赖管理 / CI/CD（2 道）

---

## 一、Feature 迭代类

### F-01 [Feature] 蜂箱档案的就地编辑、迁场与退役管理
当前 `蜂箱档案` 表格里的蜂箱只能新增和查看，没有覆盖蜂箱真实的生命周期。请在表格的每一行末尾追加 `编辑`、`迁场`、`退役` 三个操作按钮：点击 `编辑` 时该行整体切换为可编辑的输入框，并出现 `保存 / 取消` 按钮；点击 `迁场` 弹出对话框（不要用浏览器原生 confirm），让用户选择目标蜂场并填写迁场原因，提交后该蜂箱的所属蜂场字段更新并记录迁场日志；点击 `退役` 同样弹对话框确认后将蜂箱状态标记为已退役，前端默认隐藏，但表格顶部要新增 `显示已退役蜂箱` 开关，开启后退役蜂箱以灰色斜体样式回显并提供 `复用启用` 按钮。整套流程要在不刷新页面的前提下让蜂农直接在前端验证。

### F-02 [Feature] 蜂箱列表的分页、排序与多维筛选
当蜂场规模超过 50 箱后，当前一次性把蜂箱全拉到前端展示已经不合适。请在 `蜂箱档案` 表格上方新增一行工具条：左侧是搜索框（按蜂箱编号 / 备注模糊匹配），右侧是每页条数下拉框（10/20/50）、所属蜂场下拉、群势等级（弱/中/强/特强）多选标签；表格底部增加分页器，显示当前页码、总页数和总记录数;同时支持点击表头的 `编号`、`建箱日期`、`最近巡检日期`、`群势` 列在升序、降序、不排序三种状态之间循环切换，列头要用箭头图标直观体现当前排序方向。后端接口需支持 `keyword`、`apiary_id`、`strength_in`、`page`、`size`、`order_by`、`order_dir` 这些 query 参数，并在响应里返回总数。

### F-03 [Feature] 蜂箱新建接口改造为 JSON Body + Pydantic 严格校验
目前新建蜂箱用的是 query 参数提交，这违反 REST 风格，也无法承载较长的备注文本。请把它改造为接收 JSON Body，使用 Pydantic 模型严格校验：编号必填且符合 `[A-Z]\d{3,6}` 格式、所属蜂场 ID 必填且存在、建箱日期不晚于今天、备注 ≤ 500 字符，超出范围要返回 422 并把字段级错误信息带回前端。前端表单要随之改为 `application/json` 提交，并在编号输入框下方实时展示格式校验状态、在备注下方实时展示当前字数 / 上限；提交失败时把后端返回的字段级错误信息显示在对应输入框下方红色提示中，而不是一句笼统的 `添加失败`。

### F-04 [Feature] 蜂箱档案的业务字段扩展与前端呈现
现在的蜂箱表只有编号和备注两个字段，根本承载不了蜂场真实业务。请在数据库表中新增 `bee_species`（蜂种枚举：意蜂 / 中蜂 / 黑蜂 / 卡尼鄂拉蜂）、`box_type`（箱型：朗式 / 中标 / 卧式）、`location`（GPS 坐标，浮点对）、`queen_birth_date`（女王蜂出生日期）、`strength_level`（群势等级 1-5）五个字段，提供平滑的数据库迁移方案保证已有数据不丢失。前端表单要新增对应的下拉、坐标拾取（带地图小窗口或手动经纬度输入）、日期选择器、星级控件；表格要把关键字段展示出来，群势用 5 颗小蜜蜂图标渲染；同时上方筛选条增加 `按蜂种过滤` 和 `按群势过滤` 两个能力。

### F-05 [Feature] 传感器轮询接口的参数化与现场调试面板
当前模拟的传感器拉取接口（对应 `/api/slow`）延迟是写死的随机区间，无法演示不同采集策略。请把它改造为接受 `poll_interval_ms`、`jitter_percent`、`timeout_ms` 三个参数（都给合理的默认值和上下限校验），并在前端 `蜂箱传感器调试` 卡片里把传感器拉取按钮升级为一个折叠面板：展开后有三个滑块控件分别调整采集间隔、抖动百分比和超时阈值，下方实时展示一个"预计耗时区间"提示；点击触发后把每一次的实测耗时以横向条形图的形式累加到面板底部最多 20 条历史中，超过超时的请求以红色条形高亮。

### F-06 [Feature] 蜂群异常事件接口的多类型矩阵化
当前的异常模拟接口只能模拟单一异常，无法演示蜂场真实的多种突发场景。请把它改造为接受 `event_type` 参数，支持 `swarm`（分蜂）、`absconding`（弃巢逃群）、`robbing`（盗蜂）、`varroa_outbreak`（蜂螨爆发）、`queenless`（失王）、`starvation`（饥饿）、`overheating`（高温）、`flooding`（水淹）共八种事件，每种返回贴近真实业务语义的 JSON 错误体（含建议处置动作）。前端把原有的错误按钮升级为一个事件类型网格选择器：每个事件是一张小卡片带蜂业图标，鼠标悬停时显示该事件的中文成因解释，点击后触发对应事件并把响应展示在 `Last Event` 区域；同时在网格上方维护一个该会话内各事件触发次数的小型柱状图。

### F-07 [Feature] 前端内嵌的蜂场 PromQL 查询面板
现在养蜂员要看真实指标必须跳到 9090 端口，跨系统体验太割裂。请在前端新增一张卡片 `蜂场指标查询`：内含一个 PromQL 输入框（带语法高亮和最近 5 条历史下拉），右侧一个 `执行` 按钮和时间范围选择器（5m / 15m / 1h / 24h）；前端通过后端代理接口请求 Prometheus 的 `/api/v1/query_range`，把返回的时序数据用一个简单的折线图渲染出来（可使用 recharts 或自绘 svg），并按 hive_id 自动着色分组。同时预置四个快捷按钮：`蜂群活跃度`、`蜂箱重量趋势`、`蜂箱温度异常`、`累计采蜜量`，点击即可填入对应 PromQL 并执行。

### F-08 [Feature] 大型蜂场传感器并发上报压测面板
蜂场一旦扩展到 1000 箱规模，传感器并发上报对系统的压力骤增，缺少直观的压测演示。请新增一个全屏的 `蜂群压测中心` 页面：支持配置目标接口（巡检 / 传感器上报 / 异常事件）、模拟蜂箱数（1~1000）、每箱上报次数（10~10000）、上报间隔（毫秒），点击开始后用 Web Worker 在浏览器内发起请求，实时展示进度条、累计成功 / 失败数、QPS、平均耗时、P95 耗时，并把请求情况画成滚动的折线图。结束后给出一份可下载的 JSON 报告，并提供 `导出截图` 按钮（html2canvas 即可），方便提交给运维归档。

### F-09 [Feature] 蜂场自定义业务指标埋点
现有 Prometheus 抓取的全是 HTTP 标准指标，蜂场真实业务侧的 Counter / Gauge / Histogram 完全缺失。请在后端新增三个自定义指标：`apiary_harvest_kg_total`（Counter，采蜜累计公斤数）、`hive_active_count`（Gauge，当前在养蜂箱数）、`hive_inspection_duration_seconds`（Histogram，单次巡检耗时分布），并各自暴露一个触发接口（如 `/api/harvest/record`、`/api/hives/state-change`、`/api/inspections/finish`）。前端新增 `蜂场业务指标` 卡片，提供三个分组按钮触发对应指标变化，并通过轮询 Prometheus 把当前指标值实时折线展示，让蜂农直观看到自己操作产生的指标曲线。

### F-10 [Feature] WebSocket 实时推送蜂箱传感器数据
HTTP 轮询查 Prometheus 延迟较大，无法满足蜂场现场的"看到一秒前数据"的诉求。请在后端引入一个 WebSocket 端点 `/ws/hives/{hive_id}/telemetry`，由后端每秒采集（或模拟）一次该蜂箱的温度 / 湿度 / 重量 / 出入巢频次，并推送到所有订阅的前端。前端在页眉的 `Backend Online` 徽章右侧新增一个迷你 sparkline 折线（最近 60 秒蜂场总活跃度）；点击徽章可弹出一个抽屉，可在抽屉里选择某个具体蜂箱看完整的四条实时曲线，并支持暂停 / 继续推送、保留时长选择（1min / 5min / 10min）。WebSocket 断开时要带指数退避自动重连，并在 UI 上给出连接状态提示。

### F-11 [Feature] 蜂场后端日志接口与前端日志查看器
现在调试 backend 只能去 docker logs 查看，养蜂员现场根本看不到系统在干什么。请在后端用一个环形缓冲区保留最近 500 条 logging 输出（含时间戳、级别、logger 名、消息体），并暴露 `GET /api/logs` 支持按级别（DEBUG/INFO/WARNING/ERROR）和关键字过滤；同时提供一个 SSE 端点 `/api/logs/stream` 实时推送新日志。前端在底部新增可折叠的 `系统日志面板`，默认折叠，展开后是一个等宽字体、按级别上色（INFO 蓝、WARNING 黄、ERROR 红）的滚动列表，支持暂停滚动、关键字高亮、一键复制全部，方便现场把异常日志发给技术支持。

### F-12 [Feature] 蜂场设备健康检查与拓扑可视化
当前页面只有一个静态的 `Backend Online` 字样，看不到蜂场各个子系统真实状态。请在后端新增 `GET /api/health` 接口，逐项探测 DB（执行 SELECT 1）、Prometheus（curl /-/healthy）、传感器网关心跳、磁盘容量、内存占用，每项返回耗时和状态。前端把首页页眉的徽章升级为一个 `蜂场健康` 浮层：用一张 SVG 拓扑图把 Web 控制台、API 服务、PostgreSQL、Prometheus、传感器网关用线连接起来，每个节点用呼吸灯效果展示绿色 / 黄色 / 红色三种状态，点击节点弹出当前节点的探测详情，整个页面每 10 秒自动刷新一次健康状态。

### F-13 [Feature] 国际化（简体中文 / English / Español）切换
养蜂业是全球生意，蜂蜜出口国家众多，当前所有文案硬编码为中文，海外蜂农和买家用不了。请引入 i18next（或类似方案）对整个前端做国际化改造，至少覆盖所有按钮、卡片标题、表头、错误提示、空状态文案，并把蜂业专业术语（如"群势"、"分蜂"、"巢框"、"王台"）建立标准翻译表。在页眉右上角新增语言切换下拉框，切换后立即生效并把选择持久化到 localStorage；同时前端日期 / 时间格式、数字千分位、重量单位（公斤 / 磅）都要随当前 locale 改变。注意 toast 提示、表单校验错误信息也要从国际化资源里取，不能漏。

### F-14 [Feature] 亮色 / 暗色 / 跟随系统主题切换
当前页面是固定的暗色科技风格，但养蜂员在户外强光下查看屏幕需要亮色主题。请引入 Tailwind 的 darkMode 机制，把所有写死的 `bg-slate-900`、`text-slate-100` 等改造为可切换；新增一套与现有暗色等价审美的亮色"蜂蜜主题"（米白背景 + 蜂蜡黄高亮 + 蜂巢六边形纹理水印）。在页眉新增一个三态切换按钮（亮 / 暗 / 跟随系统），切换后所有卡片、按钮、表格、图表、toast 都要正确换色，颜色对比度要满足无障碍要求；选择持久化到 localStorage，并在首屏渲染阶段避免主题闪烁。

### F-15 [Feature] 响应式移动端布局与现场快捷操作
养蜂员是典型的"双手沾满蜂胶"用户，移动端必须好用。当前布局只在 md 断点做了简单网格切换，375 宽度下卡片会挤成一团、表格还会横向滚出去。请整体重排移动端：页眉的副标题和健康徽章在小屏上折叠进汉堡按钮，点击弹出全屏抽屉菜单；模拟器卡片的按钮在小屏改成纵向堆叠并放大触控区到 48px 以上；蜂箱档案表格在小屏改成卡片列表样式（每箱一张迷你卡片，左侧显示状态色块、右侧显示快捷"开箱巡检"按钮）；最后用 360px / 768px / 1280px 三个断点截屏验收，所有元素都不允许溢出。

### F-16 [Feature] 通用蜂场 API 调试 Playground
对接 IoT 厂商和第三方蜂业系统时，技术支持需要一个内置的请求调试工具。请在前端新增一张 `API 调试台` 卡片：用户可以自由输入 Method（GET/POST/PUT/DELETE/PATCH）、URL（带 baseUrl 自动补全建议）、Headers（key-value 多行）、Query Params（key-value 多行）和 Body（JSON 编辑器带格式化按钮），点击发送后展示响应状态码、响应耗时、Response Headers、Response Body（JSON 高亮折叠）。最近 20 次请求保留为左侧的历史列表，点击可一键回填，再加一个 `导出为 cURL` 按钮方便交给后端同事复现。

### F-17 [Feature] 蜂箱温度直方图的可视化呈现
P95 / P99 这类统计概念对蜂农不友好，但温度分布对他们来说极其关键。请在前端新增 `蜂箱温度分布` 卡片，展示一个真实的直方图：通过后端代理调用 Prometheus 拿到 `hive_temperature_celsius` 的桶数据（或对原始温度采样分桶），按 26~40℃ 划分 0.5℃ 一桶，把每个桶的样本数画成一根柱子（横轴是温度区间、纵轴是出现次数），并在柱子上方标出 P50 / P95 / 最高三条垂直参考线和对应数值，外加一段红色的"危险温度带"背景色（>38℃）。卡片顶部支持选择目标蜂场 / 蜂箱和时间窗口（1h / 24h / 7d），切换时图形要带过渡动画平滑重绘。

### F-18 [Feature] 蜂群异常告警规则配置与告警中心
项目里完全没有告警这一环，蜂群异常往往晚发现一周。请在 `prometheus.yml` 上挂载告警规则文件 `alerts.yml`，预置四条规则：蜂箱温度 > 38℃ 持续 5 分钟、蜂箱重量 1 小时跌幅 > 2kg、单蜂箱出入巢频次跌至 0 持续 30 分钟、连续 7 天未巡检；并在 docker-compose 中加入 alertmanager 容器接收告警（webhook 模式即可）。后端新增 `/api/alerts` 代理 alertmanager 的当前活动告警；前端新增 `蜂场告警中心` 卡片，以红 / 黄 / 蓝三色徽章列出当前 firing / pending / resolved 告警，点击徽章弹出详情抽屉显示触发蜂箱、表达式、当前值和建议处置动作。

### F-19 [Feature] 蜂箱档案与采蜜批次的 CSV 导入导出
中大型蜂场一般有自己的传统 Excel 台账，单条录入效率太低。请在 `蜂箱档案` 卡片新增 `导入 CSV` 和 `导出 CSV` 两个按钮：导出会把当前筛选条件下的全部蜂箱打包成 UTF-8 BOM 的 CSV 直接触发下载，列含所属蜂场、编号、蜂种、群势、建箱日期、最近巡检日期；导入支持拖拽或点击选择文件，前端做格式预校验（列名匹配、长度限制、必填、蜂场 ID 是否存在）后弹出一个预览对话框列出"将新增 N 条 / 将更新 M 条 / 错误 K 条"，确认后分批调用后端接口入库，并展示进度条；同时表格支持复选框多选，多选后顶部出现批量迁场 / 批量改群势操作条。

### F-20 [Feature] 蜂群异常事件的 Top10 排行榜
当蜂场告警频繁触发时，蜂农需要知道是哪些蜂箱反复出问题以便针对性处理。请在后端通过中间件维护一个内存中的异常事件记录器（保留最近 200 条蜂群异常事件，含蜂场 ID、蜂箱编号、事件类型、严重程度、发生时间），并暴露 `GET /api/incidents/top`。前端新增 `异常蜂箱排行榜` 卡片，默认按事件次数倒序展示 Top10，支持点击表头切换排序维度（次数 / 最近一次时间 / 严重程度），按蜂场或事件类型过滤；每行点开可以看该蜂箱的完整异常时间线；同时卡片右上角有一个总开关，可以临时把记录阈值从"全部记录"调整为"仅严重以上 / 仅本月"并即时生效。

## 二、0-1 模块构建类

### Z-01 [0-1] 多角色账号与权限模块
模块覆盖场主、养蜂员、技术员、外部审计四类角色的账号管理与鉴权。账号体系记录用户名、加密后的密码、角色、可访问的蜂场范围与创建时间；提供注册、登录、查看当前用户信息、退出四项能力，登录后下发短期访问凭证并支持续期。前端含登录页与注册页（带表单校验与密码强度提示），未登录访问其他页面自动跳转登录；登录后页眉右上角展示用户名、所属蜂场与退出按钮；前端请求层自动注入访问凭证，并在凭证过期时自动跳回登录；不同角色在主控台看到的操作按钮按权限隐藏。

### Z-02 [0-1] 蜂箱操作审计日志模块
